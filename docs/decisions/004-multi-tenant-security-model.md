# ADR-004: 多租户安全模型 (Multi-Tenant Security Model)

> 状态: 草案 - 待团队评审
> 创建日期: 2026-04-13
> 作者: @opus (布偶猫)

---

## 1. 状态 (Status)
已接受 (Accepted) - 2026-04-13

**评审人**: @codex, @gemini

## 2. 上下文 (Context)

系统采用 `Schema-per-tenant` 模式实现租户隔离，需要确保：
- 跨租户数据访问被严格禁止
- 原生 SQL 绕过应用层约束被阻止
- 数据库层面有强约束作为最后防线

## 3. 决策结果 (Decision)

### 3.1 租户上下文强制注入

**要求**: 所有数据库访问必须在租户 schema 内执行

**实现方式**:
```
1. 应用层: 在请求处理入口强制注入 `SET search_path TO tenant_{tenant_id}`
2. 中间件: 拦截所有数据库操作，验证 tenant_id 存在性
3. 代理层: 禁止应用代码直接构造跨租户查询
```

### 3.2 数据库强约束

**要求**: 关键表必须有租户级别的唯一约束

**约束规则**:
- 所有业务表必须有 `tenant_id` 字段
- `UNIQUE(tenant_id, business_key)` 作为数据隔离的硬约束
- 禁止使用无 `tenant_id` 的全表查询

### 3.3 SQL 注入防护

**要求**: 禁止字符串拼接 SQL

**实现方式**:
1. 使用参数化查询（Prepared Statements）
2. ORM 层禁止使用 `raw()` 执行拼接 SQL
3. 代码审查：所有 `WHERE tenant_id = ?` 必须使用绑定变量
4. 运行时检测：拦截包含 `tenant_id =` 的字面量 SQL

### 3.4 跨租户越权检测

**要求**: 测试覆盖跨租户访问场景

**测试场景**:
1. 用户 A (tenant_1) 尝试访问 tenant_2 的数据
2. 通过修改 JWT 中的 `tenant_id` 尝试越权
3. 通过构造恶意 URL 参数绕过租户验证

## 4. 后果 (Consequences)

- 数据库层面有强约束作为最后防线
- 即使应用层有缺陷，跨租户访问仍被阻止
- 性能上需要额外的 tenant_id 条件查询开销

## 5. 后续行动项

- [ ] 实现租户上下文注入中间件（SET search_path）
- [ ] 为所有业务表添加 `tenant_id` 字段和唯一约束
- [ ] 实现参数化查询拦截器，禁止字符串拼接 SQL
- [ ] 编写跨租户越权测试用例
- [ ] 编写 SQL 注入防护测试用例
- [ ] 建立数据库约束自动化检查 CI/CD 流程
